# ⚖ Legal Agent Forge

**CLI мета-агент** для автоматизации юридических процессов. Вы не пишете агента —
вы формулируете цель, а Forge *допрашивает* эту цель, устраняет неоднозначность,
фиксирует контракт ввода-вывода, **синтезирует** исполняемого агента, **тестирует**
его на реальных кейсах, **дорабатывает** при сбоях и **упаковывает** поддерживаемый
бандл, который может подхватить инженер.

Главный принцип — **уточняй прежде, чем генерировать**: обнаруживать
неоднозначность, противоречия, отсутствующие данные и расплывчатые критерии успеха —
и выносить их в виде вопросов — вместо того чтобы молча угадывать.

```
                 ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
  сырые юрид.──▶ │ 0 приём  │─▶ │ 1 уточн. │─▶ │ 2 сборка │─▶ │ 3 тест/  │─▶ │ 4 упак-  │ ──▶ бандл
  инструкции     │структура │   │+контракт │   │ синтез   │   │ доработка│   │ка+пере-  │     + spec
                 │          │   │          │   │          │   │          │   │дача инж. │
                 └──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

## Почему это не генератор шаблонов

* Он **сначала спрашивает.** Критики (детерминированные + LLM) сканируют spec на
  неоднозначность, противоречия, отсутствующие поля, расплывчатые критерии успеха
  и **конфликты пример/контракт**, и превращают важные из них в приоритизированные,
  *блокирующие* вопросы.
* Он записывает **явные допущения** вместо угадывания. В неинтерактивном режиме
  каждый неотвеченный блокирующий вопрос становится допущением с пометкой риска в
  итоговом отчёте о передаче инженеру — ничто не разрешается молча.
* Он выполняет **инженерный цикл**: тест → диагностика → доработка → пересинтез,
  пока набор тестов не станет зелёным или прогресс не остановится. Каждый прогон
  наблюдаем.
* Результат — **поддерживаемый бандл**, а не свалка кода: редактируемый промпт,
  контракт в виде JSON-схемы, конфиг, правила, тонкая стабильная точка входа,
  spec и отчёт о передаче инженеру, честно сообщающий, что ещё не проверено.

## Переиспользуемые сервисы (не переписанные заново)

| Возможность | Заимствовано из | Адаптер |
|---|---|---|
| Доступ к LLM (GigaChat, function calling, OAuth/mTLS, повторы) | `devassist` | `forge/integrations/llm.py` |
| Извлечение текста из документов (40+ форматов, маршрутизация OCR) | `extractors` | `forge/integrations/documents.py` |

Репозитории находятся через переменные окружения (`DEVASSIST_PATH`,
`EXTRACTORS_PATH`) или как соседи этого репозитория. Если учётных данных нет, Forge
переключается на **детерминированный офлайн-движок (заглушку)** (`--offline`), так
что весь конвейер — и каждый сгенерированный агент — по-прежнему работает
воспроизводимо (именно это использует набор тестов).

## Установка

```bash
pip install -e .            # устанавливает команду `forge`
# укажите на переиспользуемые репозитории, если они не соседи / не установлены:
export DEVASSIST_PATH=/path/to/devassist
export EXTRACTORS_PATH=/path/to/extractors   # репозиторий, содержащий extractors/
# реальный LLM: задайте учётные данные GigaChat (см. .env.example) или используйте --offline
```

## Быстрый старт

```bash
# 1. Опишите агента (архетип определяется автоматически). Приём + критики запускаются сразу.
forge new "Extract parties, effective date, total amount and governing law \
  from NDA and service agreements. Dates normalized. Output JSON." --name "Contract Extractor"

forge status                       # этап, открытые замечания, допущения, последний прогон

# 2. Разрешите открытые вопросы (интерактивно) — или --auto, чтобы записать допущения.
forge clarify                      # или: forge clarify --auto

# 3. Зафиксируйте контракт, поправьте поля, соберите бандл.
forge contract
forge field --add "total_amount:string:false:Total contract value"
forge build

# 4. Добавьте тест-кейсы и итерируйте до зелёного результата.
forge addtest --name nda --text "...Acme Corp and Beta LLC, effective January 5, 2024..." \
  --expect '{"parties":["Acme Corp","Beta LLC"],"effective_date":"January 5, 2024"}'
forge test --refine

# 5. Упаковка + запуск.
forge package
forge run path/to/contract.pdf --debug
```

Либо пройдите весь цикл в режиме мастера:

```bash
forge wizard "Classify incoming documents as NDA, MSA, or Other" --auto
```

## Команды

| Команда | Этап | Назначение |
|---|---|---|
| `new` | 0 | создать агента, структурировать инструкции, определить архетип, запустить критиков |
| `clarify` | 1 | интерактивные вопросы/ответы; `--auto` записывает допущения |
| `contract` / `field` | 1 | зафиксировать контракт ввода-вывода; редактировать поля вывода |
| `knowledge` / `rules` / `routing` / `checklist` | 1 | привязать базу знаний (RAG) / правила / маршрутизацию пакета / чек-лист |
| `build` | 2 | синтезировать исполняемый бандл агента |
| `addtest` / `test` | 3 | добавить кейсы; запустить; `--refine` для цикла до зелёного результата |
| `package` | 4 | выпустить spec + передачу инженеру + обновлённый бандл |
| `run` | — | запустить текущего агента (`--dry-run`, `--debug`) |
| `status` / `show` / `list` / `use` / `archetypes` | — | наблюдаемость и навигация |
| `wizard` | 0–4 | пошаговый сквозной цикл |

## Архетипы агентов (расширяемые)

`json_extraction`, `classification`, `risk_check`, `clause_comparison`,
`doc_generation`, `chat_assistant`, `workflow`, `catalog_risk_detection` (RAG по
каталогу), `rule_check` (детерминированный rule-engine), `document_package`
(обработка пакета из множества документов), `checklist` (правовая экспертиза по
каталогу проверок «да/нет» — аналог экспертиз shadow_agent).

Каждый архетип **декларативен** (`forge/archetypes/builtin.py`): значения по
умолчанию, базовые уточняющие вопросы и *стратегия среды выполнения*. Добавление —
это подкласс плюс `@register`; конвейер при этом не меняется. Архетипы разделяют
стратегии среды выполнения (`extract`, `compare`, `generate`, `chat`, `workflow`,
`rag_detect`, `rule_check`, `package_review`, `checklist_review`), интерпретируемые
в `forge/runtime/`.

### Экспертиза по чек-листу — `checklist`

Аналог промышленных правовых экспертиз (`shadow_agent/src/expertise/guarantee`):
каталог проверок «да/нет»; LLM решает, выполняется ли условие в документе (надёжная
задача), а юридический вывод (позиция / описание риска / рекомендация) берётся
**шаблоном из данных** проверки. Пример (`examples/expertise_guarantee_case/`):

```bash
forge new "Экспертиза банковской гарантии" --archetype checklist
forge checklist --file checklist.json   # [{id, question, criteria, severity, on_yes_position/finding/recommendation}]
forge build && forge run guarantee.txt
# → {results:[{check_id, verdict, position, finding, recommendation, evidence}], flagged:[...], overall_position}
```

### Обработка пакета документов (кейса) — `document_package`

Повторяет каркас промышленных юр-агентов (ср. `shadow_agent`): приём пакета
документов → классификация типа каждого по таксономии → маршрутизация → пер-
документный анализ → агрегация в плоский список находок (`flows`) + сводное
заключение + проверка комплектности (обязательные типы). Конфигурируется данными:

```bash
forge new "Ревью пакета по сделке" --archetype document_package
forge routing --file routing.json     # {taxonomy:[{code,title,description}], required:[...]}
forge build
forge run ./case_dir                  # директория = пакет документов
# → {verdict: success|partial|rejected, documents:[...], flows:[...], missing_required:[...]}
```

### Проверка по правилам — `rule_check`

Детерминированный rule-engine (без LLM): правила-как-данные с матчерами
`any`/`all`/`near` (близость)/`regex` над секциями документа.

```bash
forge new "Проверка договора по правилам" --archetype rule_check
forge rules --file rules.json         # [{id, title, severity, match, terms|pattern, window, scope, negate}]
forge build && forge run contract.pdf
```

## Сгенерированный бандл

```
agent/
  agent.yaml            # стратегия, арность входов, порог уверенности, возможности
  prompt.md             # системный промпт (редактируйте, чтобы изменить поведение)
  contract.schema.json  # контракт вывода — единственный источник истины
  rules.json            # бизнес-правила / проверки
  agent.py              # тонкая, стабильная точка входа → forge.runtime
  README.md             # как запускать, что ожидает, что возвращает
  manifest.json         # происхождение (архетип, версия, хеш схемы)
```
Плюс, в корне рабочей папки: `blueprint.json` (полное промежуточное
представление), `spec.md`, `HANDOFF.md` (что надёжно / что не проверено),
`CHANGELOG.md`, `runs/` (одна папка на каждый прогон тестов).

Каждый прогон возвращает один и тот же конверт:
`{ status: ok|partial|low_confidence|error, output, confidence, warnings, errors }`.

## Архитектура

```
forge/
  integrations/   адаптеры: llm (devassist) · documents (extractors) · stub_brain (офлайн)
  model/          промежуточное представление AgentBlueprint — цель, контракт, замечания, вопросы, допущения, тесты
  store/          хранение рабочей папки (.forge/<slug>/)
  archetypes/     декларативный реестр архетипов (точка расширения)
  clarify/        критики → замечания → приоритизированные вопросы → допущения
  pipeline/       этапы 0–4 + цикл тест/доработка, за фасадом Forge
  generate/       чертёж → исполняемый бандл
  runtime/        библиотека, которую импортируют сгенерированные агенты (стратегии, валидация, уверенность)
  cli.py          поверхность команд + мастер
```


## Офлайн / детерминированность

`--offline` (или `FORGE_OFFLINE=1`, или отсутствие учётных данных) подставляет
эвристический, не использующий сеть движок, который структурирует инструкции,
предлагает вопросы, диагностирует сбои и выполняет извлечение на основе регулярных
выражений (даты, суммы, стороны, применимое право, …). Это **воспроизводимый
запасной вариант для разработки и CI**, а не притворный LLM; при наличии реальных
учётных данных GigaChat он полностью обходится.
